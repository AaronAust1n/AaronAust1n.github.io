# [Original] When Software Becomes Handicraft, We Are Building an Information Great Depression

GitHub sees hundreds of thousands of new repositories every day. Most of them are never used, never maintained, never read. They simply exist — occupying server space, adding noise to search results, waiting to be forgotten. We talk about a “thriving open-source ecosystem”, about “record numbers of contributors”, about “democratization”. But if you open the repository list under any hot topic, what you actually see is: fifty almost identical CLI tools, twenty “weekend reimplementations” of some trendy framework, and a dozen “frameworks” whose README is three times longer than the code. These projects are not waiting for users. They are waiting to be starred once and never opened again.

My sense is this: **we are actively building an information great depression.** It’s not that information disappears, it’s that there is so much of it that it cancels itself out. A huge amount of open-source software was never written to be used; it was written to have been written. GitHub is not a code warehouse, it’s a programmer’s MySpace page.

---

This shift started from something very concrete.

In the past, if you needed to merge a bunch of PDFs, add watermarks, or batch-rename files, you would look for a dedicated tool. If the market was large enough, someone would build an app, put it on GitHub, and slowly accumulate users. That was the normal flow of the open-source ecosystem: demand drives supply, supply solidifies into tools, and tools serve the next wave of people with the same needs.

That chain is now broken.

With tools like Claude Code, building a custom tool shifted from “a few days” to “a few hours” — sometimes even ten minutes — perfectly tailored to your situation, disposable after a single use. You no longer need that GitHub project that hasn’t been updated for three years and whose Issues page is full of bug reports. This is not just an efficiency upgrade — **it is the decentralization of demand**. Long-tail needs no longer require long-tail software to catch them, because everyone can now produce their own tool on demand. No waiting for maintainers. No bending yourself around someone else’s API design.

You have a very specific need — say, parsing some company-internal log format into a report you can actually read. You used to have three options: find a half-usable open-source tool and modify it, write something from scratch, or suffer through Excel. Now there is a fourth option: spend ten minutes asking Claude Code to generate a script that fits your scenario exactly; run it, then throw it away, or toss it into a `/tools` folder and never touch it again. **Software is drifting back toward handicraft**. Everyone builds their own tools. They use them and discard them, without expecting others to maintain or adopt them.

This sounds good. The problem is: a large chunk of the open-source universe originally existed to catch precisely these long-tail needs.

---

GitHub today looks like a gigantic flea market where ninety percent of the stands sell the same low-quality product. `awesome-xxx` lists of lists, tutorial-of-tutorials, READMEs generated by Claude that are longer than the underlying code. This is not about “developers getting worse”; it’s about **incentives breaking down**. Star counts can be botted. A single citation can be added to a résumé. The sunk cost of publishing is so low that nobody has a strong reason to care about quality.

The result: truly valuable libraries are buried in the trash. You search for a feature; four of the top five hits are AI-generated clones, and the remaining one is the real thing with terrible documentation. The total volume of information keeps growing, and the cost of retrieval grows with it — and often faster.

This is the core mechanism of information inflation: **the marginal cost of producing information approaches zero, but the cost of consuming it does not fall accordingly.** You end up spending more time distinguishing signal from noise, filtering out junk, and checking whether that seemingly useful package is quietly shipping malicious code.

---

Some will say: search engines will fix this; AI summarization will fix this; community curation will fix this.

That’s an overly optimistic belief.

Search ranking algorithms have already been fighting SEO spam for two decades — now they have a new opponent: AI-generated content, produced at hundreds of times human throughput. Stack Overflow answers are increasingly polluted with AI-generated, confidently wrong solutions that still get upvotes because they *look* right. Wikipedia editors are fighting auto-generated entries. Academic databases are starting to fill with LLM-written papers that only received minimal human editing.

Filtering mechanisms are always a beat slower than production mechanisms.

---

If everyone is building their own tools, will GitHub’s “open-source garbage problem” get worse or disappear? The answer is probably: neither. What we’ll see is **stratification**.

One layer is true infrastructure — the few frameworks, toolchains, and databases that are used at massive scale. This layer will become more stable because its value is in universality, not long-tail fit. Python will not disappear, but those fifty “todo apps written in Python” will make less and less sense.

Another layer is private and disposable tools — the script you made with Claude Code, the little internal tool your company uses. These may never land on GitHub at all, or if they do, nobody needs to find them. This layer will explode in size, but it won’t clog the public commons with noise.

The real problem is the middle layer: those ambitious but mediocre open-source projects, those repos that climbed to a few hundred stars and then never saw another commit. That layer will not vanish because of Claude Code; it may actually get more crowded because “anyone can generate code now.” Barriers to entry collapse, but the cost of interpreting signals does not.

Taken to the extreme, you might ask: with information proliferating, content devalued, and work endlessly duplicated — is the internet on its way to dying?

---

So, will the internet die?

Maybe that’s the wrong question. The internet will not die, but its role as a **trusted information infrastructure** is being eroded. It will still be there; it will just become harder and harder to trust what you read on it. It may split into two modes of use. One is an internet accessed **through AI interfaces** — you don’t care what code is running underneath; you describe your need and get an answer, like turning on a tap without knowing how the plumbing works. The other is direct access to the raw layer — docs, source, data — but the number of people who operate there will shrink and professionalize.

The gap between these two modes is widening. Ordinary users are less and less exposed to the “raw materials of the internet”, and more and more to AI-processed results. This is not disappearance; it’s a layering of old and new industries: steel didn’t vanish, but most people no longer need to know how to smelt it.

So maybe the better question is: **when information is no longer trustworthy, how much value remains in the network connection itself?**

Those throwaway Claude Code scripts we’re building now are, at least, real — they solve your problem, and you watch them run successfully with your own eyes. It’s a strange kind of regression: from relying on public information infrastructure back to relying on what you can personally verify. The communal, shared, accumulable knowledge — the very promise the early internet made — is becoming harder to sustain.

This is not the death of the internet. It is the future the internet once promised, quietly souring in its own hands.

It took about two days, then I published it to my company’s plugin marketplace and open-sourced the prototype. Of course, nobody used it — because nobody even knew it existed.
