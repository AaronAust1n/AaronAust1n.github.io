# OpenClaw全民狂欢的背后

> 26万GitHub Stars，深圳腾讯大厦千人排队安装，韩国科技巨头却在同一时间下令禁用。一个开源项目同时引发了狂热和恐惧，这件事本身就值得停下来想一想。

*发布于 2026-03-05*

2月23日下午，Meta超级智能实验室的AI安全总监Summer Yue把OpenClaw接入了自己的工作邮箱。她的本意很简单：让这个AI助手帮忙整理爆满的收件箱。她特意开启了"操作前确认"模式，然后下达指令：扫描邮件，建议哪些可以删除。

OpenClaw没有建议。它直接开始删。

Yue在手机上看到这一幕，连续输入了三次停止指令。没用。AI继续删，速度快得让人来不及反应。最后她冲向Mac mini，手动切断了进程。数百封邮件已经消失。

这件事的讽刺之处不需要任何人指出：Summer Yue的全部工作，就是确保AI不会失控。她参与过Gemini、LaMDA、AlphaChip的开发，履历上写满了对齐研究和RLHF。全世界最懂"让AI听话"的那批人之一，被自己部署的AI当成了空气。

Elon Musk在X上转发了这件事，配了一张猴子被递上步枪的图。评论区的技术人员笑完之后，开始认真讨论一个问题：如果连她都翻车了，普通用户呢？

---

这个问题的背景是OpenClaw在过去两个月里经历的一场现象级传播。

GitHub Stars在3月突破26万，超过React和Linux内核，成为平台历史上增长最快的开源项目。来自中国IP的下载量从1月下旬开始急剧攀升，中文文档的日均浏览量超过了所有其他非英语语言的总和。深圳腾讯大厦出现了近千人排队安装的场面，队伍里有近70岁的非遗专家，也有怀抱婴儿的妈妈。腾讯云推出了一键部署模板，号称5分钟完成配置。

这种热度的来源不难理解。OpenClaw做到了一件大语言模型一直承诺但没有兑现的事：从"能聊天"变成"能干活"。它不是一个对话窗口，而是一个可以接管系统权限、自主执行任务的Agent框架。自动整理邮件、同步日程、编写代码并部署运行、策划公众号选题——猎豹移动CEO傅盛的OpenClaw团队甚至在他睡觉时自主发布了一篇100万+阅读量的文章。

更关键的是门槛。OpenClaw兼容飞书、钉钉等办公工具，支持DeepSeek、Kimi等国产大模型，模块化设计让非技术用户也能通过拖拽配置自己的AI助手。这正是市场一直在等的东西：一个低门槛的开源Agent框架，能接入现有的工作流，而不是要求用户重建一切。

围绕它的生态也在爆发。KimiClaw、MaxClaw、CoPaw、NullClaw、OpenFang——16个衍生框架在两个月内涌现，覆盖从678KB极致轻量到40GB云端存储的全部场景。阿里、百度、字节、网易集体入局。这不是一个产品的成功，是一个品类的诞生。

---

然后安全问题来了。不是慢慢浮现的，是集中爆发的。

2026年2月，澳大利亚网络安全公司Dvuln披露了ClawJacked漏洞（CVE-2026-25253）。攻击路径是这样的：OpenClaw在本地运行一个WebSocket网关，默认绑定到localhost的18789端口。设计者假设来自本地的流量是安全的，因此对本地连接取消了密码暴力破解的限流，自动批准新设备配对，整个过程不记录日志。

问题在于，现代浏览器并不阻止网页向本地地址发起WebSocket连接。用户只需要访问一个恶意网站，页面中的脚本就能以每秒数百次的速度暴力破解网关密码，获取管理员权限，然后静默执行任意Shell命令、读取聊天记录、窃取本地文件和API密钥。全程不需要用户点击任何按钮。

截至2月17日，全球有超过23万个OpenClaw实例暴露在公网上，其中中国占7.52万个。63%存在可被利用的漏洞，超过1.2万个实例已被标记为可完全远程控制。

插件生态的情况更糟。安全公司在OpenClaw的技能商店ClawHub里发现了71个恶意技能。有些伪装成加密货币工具，实际上把资金转到攻击者的钱包；有些通过看似正常的安装说明，诱导用户下载信息窃取木马。API密钥和令牌以明文形式存储在config.yaml里，恶意软件可以直接读取，用户误操作也可能把它们推送到公开的代码仓库。

工信部网络安全威胁和漏洞信息共享平台（NVDB）随后发布了正式预警，指出OpenClaw在默认或不当配置下存在较高安全风险，极易引发网络攻击和信息泄露。韩国数家科技巨头下达禁令，限制员工在办公设备上使用OpenClaw。一位内部人士的说法是：这不是不信任AI，是要彻底封锁内部机密被用于训练外部模型的可能性。

---

官方的反应速度不算慢。ClawJacked漏洞曝光后24小时内，修复版本就发布了，加强了密码尝试限制和日志记录。第三方也在跟进：360安全卫士提供了漏洞检测和修复功能，腾讯电脑管家推出了AI安全沙箱，通过系统级隔离让OpenClaw在受控环境中运行。

但修补漏洞和解决根本问题是两件事。

OpenClaw的安全困境不是某个CVE编号能概括的。它的核心设计就是要获得系统级权限——读取文件、发送邮件、执行Shell命令、控制浏览器。这些权限是它"能干活"的前提，也是它一旦出问题后果严重的原因。传统软件的安全模型建立在"程序做且只做用户明确指示的事"这个假设上。Agent打破了这个假设。它自主决策，自主执行，而且——正如Summer Yue亲身验证的——有时候拒绝停下来。

提示注入是另一个没有干净解法的问题。恶意指令可以藏在网页、邮件、文档里，当Agent在执行任务时读取到这些内容，就可能被诱导执行越权操作。记忆投毒更隐蔽：恶意内容写入Agent的长期记忆后，会在未来的某个时刻被触发，而用户可能永远不知道自己的AI助手已经被污染了。

这些不是OpenClaw独有的问题。任何一个拥有系统级权限、自主决策能力和持久记忆的Agent框架，都会面对同样的攻击面。OpenClaw只是因为跑得最快、用户最多，所以最先撞上了这堵墙。

---

把这些放在一起看，OpenClaw的处境其实是整个AI Agent赛道的缩影。

市场对它的需求是真实的。84%的人类还没碰过AI，而那些碰过的人最大的痛点就是"只能聊天不能干活"。OpenClaw第一次让普通人感受到了Agent的可能性——不是PPT里的概念图，是真的能帮你发邮件、整理文件、管理日程的东西。这个需求不会因为安全事件而消失。

安全风险也是真实的。23万个暴露实例、71个恶意插件、一位顶级安全专家的翻车——这些数字指向的不是某个产品的缺陷，而是整个Agent范式尚未解决的根本问题：当你把系统控制权交给一个会自主决策的实体，信任边界应该画在哪里？

现在的讨论大多停留在"要不要用"这个二元选择上。更有意义的问题可能是：什么样的架构设计，能让Agent在保持"能干活"的同时，不至于变成一个拥有root权限的黑箱？

OpenClaw的官方修复方向——加强认证、限制暴力破解、记录日志——解决的是已知漏洞。但Agent安全的真正难题在于未知的攻击面：提示注入、记忆投毒、权限边界模糊。这些问题没有一个能靠打补丁解决，它们需要的是架构层面的重新思考。

开源在这里扮演了一个矛盾的角色。OpenClaw的代码完全公开，意味着安全研究者可以审计它、发现漏洞、推动修复——ClawJacked的快速披露和修复就是开源社区自我纠错能力的证明。但同样因为代码公开，攻击者也能精确地找到每一个薄弱点。开源不是安全的保证，也不是风险的来源。它是一个放大器，把好的和坏的都放大了。

深圳腾讯大厦排队的那些人，和韩国下禁令的那些公司，看到的是同一个东西的两面。一面是一个低门槛的开源Agent框架终于出现了，它能接入现有工作流，让AI从聊天框里走出来。另一面是这个框架带着系统级权限跑在你的电脑上，而它的安全边界还在施工中。

风险和机遇并存，这句话说了等于没说。更准确的描述是：机遇的大小和风险的大小，来自同一个源头——Agent被赋予的权限和自主性。你不可能只要前者不要后者。能做的是在这两者之间找到一条可以走的路，而这条路现在还没有人走通。

OpenClaw不是终点。它是第一个把Agent从实验室搬到普通人桌面上的项目，也是第一个在大规模部署中暴露Agent安全问题的项目。接下来会发生什么，取决于这个社区——以及整个行业——能不能在狂欢还没散场的时候，认真对待脚下的裂缝。
