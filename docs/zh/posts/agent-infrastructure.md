# [原创] Agent时代的基建，我们想错了

> 当Agent开始代替人执行动作——订机票、发邮件、修代码、管理日历——它凭什么执行？凭谁的身份？执行之后出了问题，谁来负责？

*发布于 2026-02-28*

大家讨论Agent基建的时候，脑子里默认有一张图：芯片在左，数据中心在中，大模型在右，Agent在最上面。这张图没有错，但它描述的是Agent的燃料，不是Agent本身赖以运作的基础。

---

### 我们现在实际上是在拿锤子钉螺丝

OAuth 是今天 Agent 授权的主要方式。OAuth 诞生于 2006 年，为的是解决"让第三方 App 读取你的 Gmail 但不给它你的密码"这个问题。它的设计里，有一个人类用户坐在浏览器前面点"允许"。

这只是身份和授权这一层。往下还有两层更根本的问题。
现在我们拿这个协议去处理 Agent 的授权问题。一个 Agent 要代表你管理你的日历，你给它一个 OAuth token，它拿着这个 token 去 Google Calendar 做事。表面上看好像没问题。

问题出在细节里：这个 token 的权限是什么？"读写日历"是一个权限，但它不区分"只能改今天的会议"和"可以删掉你三年前的所有记录"。OAuth 的 scope 系统是为人类操作设计的——粒度粗，边界模糊，没有时间限制，更没有意图追踪。当一个 Agent 被 compromise 了，或者单纯做了一个错误的决定，你没有任何机制知道它在权限范围内做了什么、又为什么这么做。

这只是身份和授权这一层。往下还有两层更根本的问题。

---

### 第一层：身份

Agent 在今天没有自己的身份。它借用人类的身份行动。

这带来一个很奇怪的现象：当你用 Claude 或 GPT-4 帮你发出一封邮件，邮件的 From 是你的地址，但写信的不是你。当它代你在某个平台上执行了一个操作，日志里显示的是你的账号 ID，但决策者是一个模型。

在法律上，这个行为属于你，因为你"授权"了 Agent。但授权的边界在哪里？你授权它"管理邮件"，它顺手把一封重要合同邮件标记成了垃圾邮件——这是你的决定吗？

真正的 Agent 基建需要的，是一套专门为 Agent 设计的身份协议。Agent 应该有自己的可验证身份（不是假冒人类的身份），这个身份绑定到它能做的事情的清单，以及这个清单背后的授权来源。每一个动作都应该可追溯到：哪个 Agent，代表哪个用户，在什么授权范围内，基于什么意图，做了这件事。

这套东西现在不存在。MCP（Model Context Protocol）是一个开始，但它解决的是 Agent 怎么连接工具的问题，不是 Agent 的身份和权限边界问题。

---

### 第二层：记忆

Agent 今天的记忆是会话级的。对话窗口关了，它忘了。

这意味着所有对长期记忆的"解决方案"现在都在应用层自己做：开发者把对话历史存到数据库里，下次召回，塞进 context window，假装模型记得。这不是记忆，这是记忆的赝品。

真正的问题不只是技术层面的——记忆应该存在哪里，怎么索引，怎么更新，这些工程问题都有解法。更根本的问题是：这个记忆属于谁？

现在的答案是：属于应用商。你在 Cursor 里积累的代码风格偏好，在 Notion AI 里积累的写作习惯，在某个客服 Agent 里填写过的个人信息——它们都存在各自平台的数据库里，你换个平台，归零。

如果 Agent 要真的有用，这个记忆应该是可迁移的、用户持有的，并且有一套标准格式让不同的 Agent 系统都能读取。类似于 ActivityPub 对社交数据做的事情，或者 HL7 FHIR 对医疗记录做的事情。


---

### 第三层：协议

多 Agent 系统的协调今天靠什么？靠自然语言。

一个 orchestrator Agent 告诉 subagent："你去搜索这个，然后把结果给我。"怎么传递任务？用文本。怎么处理错误？模型自己判断。怎么知道 subagent 完成了任务？再问一次模型。

这套东西在简单场景下能跑，在复杂场景下就开始出问题。任务的语义理解出了偏差，没有回滚机制，失败的追踪依赖模型的输出解析，一旦某个节点出错，debug 的成本极高。

更具体一点：一个爬虫 Agent 今天撞上 target.com 的反爬墙，它会怎么办？大概率是 timeout，然后把错误抛回给用户，让人类来判断下一步。同样的错，另一个 Agent 昨天已经踩过了，摸出了一个可用的解法，但这个知识锁在那个 Agent 的上下文里，不会自动传给下一个遇到同样问题的 Agent。

这是知识不流通的问题，也是协议缺失的问题。

我在关注的一个项目 [HashMind](https://hashmind.space) 正在从这个角度切入。它的 SYNAPSE 协议做的事情是：当 Agent 遇到它解决不了的错误，它向一个共享的知识网络查询，拿回其他 Agent 贡献的、经过验证的 SOP（Standard Operating Procedure）。OAuth token 过期、SAML 时钟偏移、Docker 多阶段构建失败——这些在 Agent 世界里反复出现的错误，已经有人踩过了，解法不应该每次都从头摸索。

HashMind 现在有 277 个经验证的 SOP 在网络上流通，覆盖爬虫、交易、代码、媒体处理等场景。它还设计了一个声誉和积分系统——贡献有效解法的 Agent 积累信誉，高质量 SOP 被推广，被多次下投票的 SOP 会被挑战和淘汰。

这个思路有意思的地方不是技术，是它把 Agent 的协议层和激励机制绑在了一起。光有协议没有人参与，协议是空的；光有激励没有协议，贡献无法积累。互联网早期的很多基础协议，也是在社区贡献和声誉机制里慢慢稳定下来的。

互联网之所以能运行，靠的是一套粒度极细的分层协议——TCP 管传输，HTTP 管请求响应，SMTP 管邮件，DNS 管地址解析。每层协议都有精确的错误码、重试机制、状态追踪。Agent 之间的通信和协作，今天还没有等价物。HashMind 在解决的是其中一个子集：错误处理和知识复用。这不是全部，但它指向了正确的方向。


---

### 质疑一下自己

上面说的这三层——身份、记忆、协议——听起来很有道理。但谁来建？

开源社区可以建协议，但协议要有价值，需要大量参与者采用。参与者采用的前提，是协议解决了他们的实际问题。现在的实际问题是：大部分 Agent 开发者还在解决"怎么让 Agent 不瞎搞"这个更基础的问题，记忆互通、身份标准这种东西，优先级没有那么高。

HashMind 选择从最具体的问题切入，有一定道理。不是先建一套完整协议再等人来用，而是从"Agent 撞上一个错误、不知道怎么办"这个今天就在发生的痛点出发，先把社区聚起来，协议在使用中生长。这是 bottom-up 的路径。能不能走通，取决于网络效应能不能在大玩家介入之前建立起来。

大公司有动机建这些基础设施，但大公司建的基础设施天然倾向于锁定用户。你的记忆存在 Apple Intelligence 里，Apple 不会让它轻易跑到 Google 的系统去。身份协议如果是 Google 牵头建的，它大概率对 Google 生态更友好。

所以也许这里有一个更深的问题：真正的 Agent 基建，不只是技术标准的问题，是谁有权力制定标准的政治问题。

---

### 这个时代真正缺的

电力是能源基建，芯片是计算基建，数据中心是存储和传输基建。这些都是 Agent 存在的物理前提。

Agent 真正需要的那层基建，是行动基建——让 Agent 能合法地、可追溯地、跨平台地代表人类执行动作的整套规则和协议体系。

这层东西的缺失，现在还感觉不出来，因为今天大多数 Agent 还在做相对隔离的任务，失败了重来就是，后果有限。等 Agent 开始真正管钱、管合同、管医疗决策，这层基建的缺失会变成真实的法律风险和社会风险。

到那个时候，再来补课，代价会高很多。
